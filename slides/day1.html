Coding Paradigms: From Vibe to Custom Pipelines

The world of AI in software development is evolving rapidly. To understand where we're headed, it's helpful to see where we've come from and the different philosophies at play today.

1. Vibe Coding (e.g., Replit, Firebase Studio)

**Analogy:** The "all-in-one" workshop. Everything you need is in one place, perfectly configured to work together.

**Description:** A highly integrated, low-friction environment where a platform provides the IDE, runtime, deployment, and even a database in a single package. The goal is to remove all the boilerplate and setup so you can "vibe" and focus purely on the code.

  • Pros: Incredibly fast setup, minimal friction, perfect for prototyping and small apps.
  • Cons: Vendor lock-in, limited customization, less control over the underlying stack.
  • Use Case: Quick demos, educational projects, or small-scale applications where you value speed over control.

2. AI-Assisted Coding (e.g., VS Code Copilot, Cursor)

**Analogy:** A developer with a brilliant co-pilot.

**Description:** This is the most common paradigm today. You use your own local IDE and retain full control over your environment, but AI is integrated directly into your workflow to augment your skills. The AI provides powerful, context-aware suggestions, code completions, and debugging assistance.

  • Pros: Augments developer skills, doesn't replace them. Full control over your IDE, tools, and codebase. Powerful for boilerplate, debugging, and learning new APIs.
  • Cons: The AI is reactive, not proactive. You still need a solid understanding of the code to validate the AI's output.
  • Use Case: Everyday coding, learning new frameworks, and improving code quality.

3. Custom AI-Augmented Pipelines (Our Focus)

**Analogy:** A highly specialized, automated factory designed and built by engineers.

**Description:** This is the core philosophy of this program. It's about building custom, modular, and repeatable AI-driven workflows for the *entire* SDLC. This goes beyond a single AI assistant in an IDE. It's about using AI to orchestrate, validate, and automate across different tools and phases of development, creating a custom pipeline that is a competitive advantage.

  • Pros: Maximum flexibility, full control, and a significant competitive advantage. Highly tailored to your specific project and organizational needs.
  • Cons: Steeper learning curve, requires a deeper understanding of AI frameworks and DevOps principles.
  • Use Case: Automating requirements, generating CI/CD pipelines, performing automated security red teaming, and building multi-agent research teams.

Our program is focused on mastering this third paradigm, using AI-assisted coding as the foundational skill to build these powerful, custom pipelines.

Environment Setup: Getting Ready to Code

Before we dive into our first lab, it is crucial to set up a clean and consistent development environment. We will use a Python virtual environment to manage our project's dependencies, ensuring a reproducible environment and preventing conflicts with other projects on your machine.

Step 1: Create a Virtual Environment

Open your terminal and run the following command. This creates a new directory named `venv` that contains an isolated Python environment. 

python -m venv venv

Step 2: Activate the Virtual Environment

Before you install any packages, you must activate the environment.

For macOS/Linux: 

source venv/bin/activate

For Windows: 

.\venv\Scripts\activate

You'll know it's active when your terminal prompt is prefixed with `(venv)`.

Step 3: Install Dependencies from `requirements.txt`

The `requirements.txt` file is a manifest of all the Python libraries and their versions that our project needs. To install them in your new environment, run this command: 

pip install -r requirements.txt

You can verify the installation by running `pip freeze`, which will show you a list of all the installed packages and their versions.

Assisted Practice: AI-Powered Requirements & User Stories

Lab Overview: In our first hands-on lab, you will act as a Product Manager. Your mission is to use an LLM as a co-pilot to take a vague, one-sentence problem statement and transform it into a structured, machine-readable requirements document. We will use AI to brainstorm features, identify user personas, and generate detailed Agile user stories with acceptance criteria.

(Detailed instructions are in: `D1_Lab1_AI_Powered_Requirements_User_Stories.ipynb`)

Python Libraries & Concepts for this Lab ▼
  • os, sys: Standard Python libraries for managing file paths.
  • json: Python's built-in library for working with JSON data, which will be our target output format.
  • Helper Functions (`utils.py`):
    • setup_llm_client(): Configures the API client for our chosen LLM.
    • get_completion(): Sends a text prompt to the LLM and retrieves its response.
    • save_artifact(): Saves generated outputs to our project's `artifacts` directory.
Challenge 1 (Foundational): Brainstorming Features & Personas ▼

Task: Use a simple, open-ended prompt to brainstorm potential features and user personas for our new application.

Key Concepts:
  • User Personas: Fictional characters created to represent different user types.
  • Open-Ended Prompting: Giving the LLM creative freedom to generate ideas.

Key Learning: This step shows how AI can act as an instant brainstorming partner, providing a broad set of ideas to build from.

Challenge 2 (Intermediate): Generating Formal User Stories ▼

Task: Now, create a more structured prompt to convert the brainstormed ideas into formal Agile user stories with detailed acceptance criteria in Gherkin format (`Given/When/Then`).

User Story Format:
  • "As a [user], I want [a goal], so that [a benefit]."
  • Gherkin (`Given/When/Then`): Defines testable acceptance criteria for a user story.

Key Learning: This moves beyond brainstorming to structured output generation. The JSON output can be directly consumed by other tools (e.g., Jira).

Challenge 3 (Advanced): Programmatic Validation & Artifacts ▼

Task: Write a Python function to programmatically validate the AI's JSON output, ensuring it has the correct structure and is not empty, before saving it as a formal project artifact.

Key Concepts:
  • JSON Parsing: Converting a JSON string to a Python object using `json.loads()`.
  • Data Validation: Using Python code to check for required keys, data types, and non-empty lists.

Key Learning: This is the crucial step of trusting AI output. You treat the AI as an external service whose output must be validated before use, a core principle in modern AI development.

Content: AI for Formal Documentation

Now that we have our structured requirements, we can use them to generate higher-level project documentation. A well-written **Product Requirements Document (PRD)** serves as the central source of truth for the entire product team, outlining the project's purpose, features, and target users. AI can automate the creation of these documents by synthesizing our granular user stories into a cohesive, comprehensive narrative.

This process transforms the task of documentation from a manual, time-consuming effort into an automated, AI-assisted workflow, allowing teams to stay aligned and focused on delivery.

Assisted Practice: Generating a Product Requirements Document (PRD)

Lab Overview: Building on our previous lab, you will now use the structured `day1_user_stories.json` artifact to generate a formal, comprehensive Product Requirements Document (PRD). You will learn how to prompt AI to fill in a pre-defined template, ensuring consistent and complete documentation.

(Detailed instructions are in: `D1_Lab2_Generating_a_Product_Requirements_Document_PRD.ipynb`)

Python Libraries & Concepts for this Lab ▼
  • load_artifact(): Our helper function to read the user stories JSON file from the `artifacts` directory.
  • Markdown: The target output format for our PRD, using headings, lists, and bold text for clarity.
  • Templates: Providing the LLM with a structured template file to ensure the output adheres to a specific format.
Challenge 1 (Foundational): Generating a Simple PRD ▼

Task: Use the loaded user stories to generate a simple PRD with an introduction, a summary of personas, and a list of key features/user stories.

Key Concepts:
  • Data Synthesis: The AI's ability to summarize and synthesize information from a large data source.
  • Markdown Output: Generating a clean, human-readable document.

Key Learning: This shows how AI can efficiently create high-level documentation from detailed, low-level data.

Challenge 2 (Intermediate): Generating a PRD from a Template ▼

Task: Provide the AI with a formal PRD template and have it populate the document using the user stories. The AI must also creatively fill in sections not present in the original data, like "Success Metrics."

Key Concepts:
  • Structured Content Generation: Using a template to guide the AI's output, ensuring consistency.
  • Creative Inference: The AI's ability to infer logical content based on project goals.

Key Learning: This demonstrates the power of combining AI's creative intelligence with your structured templates.

Challenge 3 (Advanced): Programmatic Validation with Pydantic ▼

Task: Use an LLM to generate a Pydantic model that represents the structure of your PRD.

Key Concepts:
  • Pydantic: A data validation library that uses Python type hints.
  • Documentation as Code: Using code to define and enforce the structure of your documents.

Key Learning: This is a critical, advanced step. It introduces automated governance, ensuring that all future documents adhere to a formal, code-based standard.

Daily Wrap-up & Q&A

Key Takeaways

  • AI can transform unstructured ideas into structured, actionable requirements.
  • The most powerful AI workflows involve treating AI as an external service whose output must be validated.
  • Prompting for a specific, machine-readable format (e.g., JSON, markdown) is key to reliable automation.
  • Documentation can be a code-based, AI-driven process that ensures consistency and saves time.
